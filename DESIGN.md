# persona-suite 설계 기록

작성일: 2026-06-05
상태: 5 스킬 빌드 완료 — #1 `defining-requirements`·#2 `designing-data-model`·`validating-test-lists`(이동) 검증됨. #3 `designing-code` v1 (light 경로 dogfood 검증 / heavy 코어 미검증). `auditing-production-risks` v1 (단발 감사 경로 4회 검증·전 런 CRITICAL 0 / 재감사 루프 미검증).

개발 라이프사이클 단계별로 시니어 페르소나 티키타카를 돌려, **실제 피처를 정의/설계하면서 + 사용자가 프로세스를 체화**하는 스킬 모음. DB 스키마를 페르소나 핑퐁으로 문서화하던 선행 시스템에서 검증된 페르소나-핑퐁 패턴을, 라이프사이클 전반으로 확장한 것이다. (선행 사례: 저자의 한 프로젝트에서 운영하던 DB 스키마 문서화 페르소나 시스템 — 프로젝트 로컬 자산이라 이 플러그인은 그것에 의존하지 않고 *패턴만* 가져왔다.)

> 이식성 원칙: 이 플러그인은 cross-project / cross-machine 으로 배포된다. **특정 프로젝트의 경로·파일·수치를 플러그인 파일에 박지 않는다.** 그런 specifics 는 각 프로젝트의 `<repo>/.claude/persona-profile.md` 에만 산다.

## 1. 배경 / 문제

요구사항 정의는 "모호한 한 줄(원츠)"을 "검증 가능한 계약"으로 깎는 작업이다. 실무 가치는 (a) 기획을 현행 코드와 대조해 빈틈을 찾고, (b) 애매한 정책을 질문으로 못박고, (c) 기획서에 없는 예외·비기능 요구를 끌어내는 데서 난다. 설계 당시 저자의 스킬 생태계엔 이 단계를 강제하는 도구가 없었다 — 아이디어 탐색(greenfield)·구현 계획·테스트 리스트 검증 스킬은 있었지만, "피처는 대략 정해졌고 이제 요구사항을 엄밀히 정의"하는 단계가 비어 있었다.

## 3. #1 `defining-requirements` 구성

- 페르소나 6종: `[PM]`①요구사항 · `[REV]`②기능정의(메인) · `[TECH]`③기술설계 · `[SEC]`횡단보안 · `[OPS]`운영 · `[QA]`검증.
- 플로우 게이트 5종: 스코프 슬라이싱(자기 완결 조각 분리 — tier 는 본체 기준) / right-size tier / 수정 시 AS-IS 코드 조사(일급 step) / 단방향+역류 / 종료 완성도 패스.
- 내용 소스: 실무 기능개발 사이클 방법론(①②③ 구분·단방향·NFR표·Saga·expand-contract·right-size)을 제네릭화해 흡수 (런타임 파일 의존 ❌ — 원본이 어느 프로젝트에 있든 플러그인은 내용만 내장).
- 상세는 `skills/defining-requirements/{SKILL.md, personas.md, output-template.md}`.

### 페르소나 6종이 된 이유 (gap 감사 결과)

`[OPS]` 는 완성도 감사에서 추가됨 — 산출물이 운영자·admin 개발자 청중인데 운영 렌즈가 없던 구멍. 그 외 보강: 지표 측정가능성(`[PM]`), 관측성 NFR(`[TECH]`), 종료 완성도 패스(`[REV]`), 수정 시 AS-IS 일급화.

### 명시적 제외 (AVOID)

회고/임시코드 cleanup 추적, 비용 렌즈, 프론트 계약 전용 페르소나, 학습 retention 루프. (요구사항 정의 스킬이 전과정 스킬로 번지는 것 방지). 코드 리뷰·테스트 리스트 감사·상세 DB 설계는 기존 도구로 핸드오프 (반기계적 → 제네릭이 맞는 도구, DRY).

## 6. dogfood 로그 (검증 이력)

> build-now-refine-later 전략의 **검증 상태 단일 출처**. 스킬을 실제 피처에 돌릴 때마다 1줄 추가 — cross-machine 으로 "뭐가 검증됐나"를 repo 안에서 본다. (프로젝트명은 익명 — 이 문서 관례)
>
> **익명화 규칙 (공개 repo)**: 프로젝트/회사 실명 금지. 반영 칼럼은 이 repo 커밋 SHA만 적는다 — 반영처가 외부(프로젝트) 문서라 SHA가 없으면 `(프로젝트 문서)` 로만 표기. 위반은 CI `masking-guard` 가 차단한다 (denylist 는 repo secret `MASKING_DENYLIST`).

### 검증자 누락 실측 판정 (감사·검수 공통)

검증자(8.5 감사·§8 종료 검수·테스트 리뷰어 등)가 "위반을 놓쳤다"는 검증 시점엔 관측 불가능 — **사후 채널 역추적**으로만 판정한다. 누락 1건 = 아래 3조건 전부 충족:

1. **시점**: 결함의 도입 커밋이 그 검증 런이 본 diff 안에 있다 (git blame/log 로 판정 — 검증 *이후* 커밋이 만든 결함은 제외).
2. **범위**: 그 검증자의 체크리스트/기준 범위 안이고, 검증 시점 기준 **CRITICAL 로 분류됐어야 할** 결함. 범위 밖이면 누락이 아니라 **목록외 갭** → checklist 역주입 회고 루프 대상 (모델 탓 아님). WARN 급은 기록만, 카운트 제외.
3. **발견 채널**: 사후 발견 — ① 운영 모니터링(에러 트래커·DLQ·정합성 스크립트) ② PR 리뷰·코드리뷰·CI ③ 후속 작업 중 발견.

기록: dogfood 로그에 1행 (발견 채널 · 카테고리 · 어느 검증 런의 누락인지 역참조). **대응은 N=1 즉시**: 누락 1건 실측 → **그 검증자의 모델을 상위 tier 로 교체**해 운용한다 (검증 품질은 추론 깊이에 민감 — 모델 교체가 가장 싼 첫 레버, 체크리스트 수술보다 먼저). 상위 모델로도 같은 검증자가 재누락하면 모델이 아니라 **체크리스트/프롬프트 결함으로 재분류** — 본문(checklist·기준)을 보강한다. tier 분담의 전제는 `LIFECYCLE.md` §모델 운용.

## 7. 한계 / 트레이드오프

- 떠먹이기 드리프트: 협업 모드가 각도를 너무 쉽게 주면 학습이 죽음 → "막혔을 때만" 보정으로 방어하나 해석 여지 남음.
- 트리거 누락: `[SEC]`/`[OPS]` 키워드 기반이라 비전형 표현 놓칠 수 있음 → 챕터 자기점검의 트리거 점검(1차) + 종료 완성도 패스(2차)로 방어 (`PRINCIPLES.md` §4). 단 1차엔 **순환성**이 남음 — 트리거를 놓친 컨텍스트가 자기 누락을 점검. 챕터마다 서브에이전트 점검은 기각(2026-06-10): 감사 대상이 외부화 산출물이 아니라 대화라서(직렬화하면 독립성 희석, 안 하면 일반론 노이즈) + 핑퐁 리듬 파괴. **종료 독립 검수 설치**(2026-06-10, `PRINCIPLES.md` §8, additive — [REV] 패스 유지): 외부화된 완성 문서 대상이라 8.5와 동형, self-contained 검증 겸용. 당초 "8.5 입증 후 이식"이었으나 선설치로 전환 — 메커니즘은 8.5에 설계 완료(만들 내용이 이미 있음) + 다음 피처의 설계가 구현(8.5 dogfood)보다 먼저 와 verify-then-port 는 한 사이클 손해 + 가역적(몇 줄). 대신 **제거 트리거**: dogfood 2회 연속 실효 없음(ⓑ[REV] 중복·ⓒ일반론뿐, ⓐ실제 누락 0)이면 제거.
- 프로필 미작성 시 마찰: 묻기로 동작하나 매번 재질문 → 자동 생성으로 1회성화.
- 체크리스트 밖 신종 리스크: 감사(8.5)는 목록 기반이라 목록 밖은 `목록외` 자유 발견에 의존(false positive 높음) → 운영(12~13)에서 사고 발견 시 checklist.md 역주입(회고 루프)으로 수렴.
