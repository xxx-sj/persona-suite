# 백엔드 기능 개발 사이클 (실무 일반)

> 일반적인 실무 기준의 기능 개발 한 사이클. 프로젝트 무관 공용 방법론 레퍼런스.
> 관통 예시: **"사용자가 적립금(포인트)으로 상품을 구매하는 기능"**
> 구체 브랜치/배포 실현(GitOps·develop→main 등)은 프로젝트별 — 이 문서는 *방법론(프로젝트 무관)* 만. 요약 맵은 [`../LIFECYCLE.md`](../LIFECYCLE.md).

---

## 0. 전체 사이클 한눈에

```
요구사항 정의        왜·누구·성공이 뭔가 (비즈니스)
  → 기능 정의        시스템이 밖에서 어떻게 동작하나 (블랙박스)
  → 기술 설계        내부를 어떻게 구현하나 (API계약·스키마/마이그레이션·트레이드오프)
  → 작업 분해        배포 가능한 작은 단위로 쪼갬
  → 테스트 리스트업   인수조건·엣지케이스에서 케이스 도출
  → 구현             feature 브랜치 + flag
  → 테스트 진행       단위·통합 (CI 자동화)
  → PR·코드 리뷰·CI   ← 머지 게이트
  → e2e 테스트        staging 환경
  → 배포             롤백 전략은 *전제* / 점진 출시(canary·flag)
  → 모니터링          메트릭·로그·트레이싱·알람
  → 이상 시 롤백 실행
  → 회고 + 정리       flag·임시코드 제거 → 다음 사이클
```

## 1. 세 단계의 경계 (핵심 멘탈 모델)

> 가장 자주 뭉개지는 구간. 같은 관심사가 단계마다 다른 언어로 변환된다.

| 단계 | 답하는 질문 | 언어 | 금지 |
|---|---|---|---|
| 요구사항 정의 | **왜·누구·성공 지표** | 비즈니스 | 해결책(어떻게) 적기 |
| 기능 정의 | **밖에서 본 동작** (입력→출력·규칙·상태) | 블랙박스 | 내부 구현 적기 |
| 기술 설계 | **내부 구현** | 기술 | 새 요구사항 만들기 |

예시 (`이중 차감 방지`가 단계마다 변하는 모습):

| 관심사 | 요구사항 | 기능 정의 | 기술 설계 |
|---|---|---|---|
| 이중 차감 | "돈이 두 번 빠지면 안 됨"(NFR) | "같은 멱등키는 1회만 처리, 동일 응답 반환" | "Idempotency-Key 유니크 제약 + 조건부 atomic update" |
| 잔액 부족 | "잔액 없으면 결제 불가" | "거부 + `INSUFFICIENT_BALANCE` + 부족액" | "`WHERE balance>=amt` update 0행이면 throw" |

**관통 규칙**: 위 → 아래 **단방향**. 설계 중 "이 경우 어떻게 하지?"가 나오면 그건 기능정의의 빈틈 → 한 칸 위로 올라가 메우고 내려온다. 설계에서 즉흥 결정하면 명세와 코드가 어긋난다.

## 2. 단계 상세 (예시: "적립금으로 상품 구매")

### ① 요구사항 정의 — 왜 만드는가

**본질**: "왜 만드는가 + 무엇이 성공인가"를 비즈니스 언어로 합의. 해결책이 들어가면 안 됨.

**산출물**
- **배경/문제(Why)**: "적립금이 쌓여만 있고 쓸 곳이 없어 결제 전환이 낮다"
- **목표 + 성공 지표**: "적립금 사용률 X%↑" — 측정 가능해야 함
- **사용자/이해관계자**: 구매자 / 정산·CS팀
- **범위 ↔ 비범위**: "전액 적립금 결제만, **카드+적립금 혼합은 비범위**" ← 비범위 못박기가 핵심
- **제약**: 법·정책(유효기간), 일정, 기존 시스템 호환
- **비기능 요구사항(NFR)** ← 기획서엔 없음, 서버 개발자가 끄집어냄:

  | NFR | 질문 |
  |---|---|
  | 정합성 | 차감+주문이 원자적이어야 하나? (돈→yes) |
  | 동시성 | 동시에 두 번 누르면? |
  | 멱등성 | 재시도로 같은 요청 두 번 오면? |
  | 성능 | P95 목표·피크 QPS |
  | 가용성 | 적립금 서비스 죽으면 주문도 막히나? |
  | 관측성 | 차감 실패를 어떻게 감지·추적 |
  | 감사/규정 | 변동 이력 보관 기간 |

> **흔한 실패**: NFR 없이 happy path만 받음 → 구현 다 하고 "동시 결제 이중 차감" 터짐.
> **시니어 포인트**: 요구사항의 모순·빈틈·암묵 가정을 질문으로 드러냄 — "적립금 결제도 환불되나? 환불하면 적립금/현금 중 무엇으로?"

### ② 기능 정의 — 밖에서 본 동작 (블랙박스)

**본질**: 시스템을 블랙박스로 보고 외부에서 관찰되는 동작을 빠짐없이 명세. 내부 구현은 아직 없음.

**산출물 (유스케이스 명세 형태)**
- **액터/사전조건**: 로그인 구매자, 잔액 보유, 재고 있음
- **기본 흐름(Happy)**: 상품·수량 선택 → 결제요청 → 잔액≥금액이면 차감 → 주문 생성 → 완료
- **비즈니스 규칙**: 최소 사용 단위, 유효기간 만료분 사용 불가, 만료 임박분부터 FIFO 차감
- **예외/대안 흐름 (기능정의의 알맹이 80%)**:

  | 상황 | 관찰 가능한 결과 |
  |---|---|
  | 잔액 부족 | 거부 + 부족액 |
  | 재고 소진 | 거부, 차감 없음 |
  | 동일 요청 재전송 | **1회만** 차감 |
  | 주문 생성 실패 | 차감 **롤백** |
  | 부분 환불 | 환불액만큼 적립금 복구 |
- **상태 전이**: 주문 `결제대기→결제완료→(취소/환불)`, 각 전이의 적립금 처리
- **입출력 명세**: 요청(상품ID·수량·사용포인트) / 응답(주문번호·차감액·잔여) / 에러 종류

> **요구사항 vs 기능정의 경계**: "이중 차감을 막아야 한다"(요구사항/NFR) → "동일 멱등키 요청은 최초 1회만 처리하고 이후엔 같은 응답 반환"(기능정의/동작). 전자는 목표, 후자는 관찰 가능한 동작.
> **흔한 실패**: happy path만 적고 예외 흐름을 비움 → 예외가 코드에서 즉흥 처리됨(일관성 깨짐).
> **시니어 포인트**: 돈이 걸린 동작은 모든 예외에 "차감은 어떻게 되나(롤백/유지)"를 못박는 게 명세의 핵심.

### ③ 기술 설계 — 내부 구현 (HOW)

**본질**: 기능정의(블랙박스)를 어떻게 구현할지 내부를 여는 단계. 새 요구사항을 만들면 안 됨(역류). 항상 대안 비교 + 트레이드오프 + 한계 동반.

**산출물**
- **(a) 컴포넌트/책임 + 시퀀스**: `OrderService` / `PointService` / 트랜잭션 경계
- **(b) 데이터 모델**: `point_balance`(잔액) vs `point_transaction`(이력 append — 감사·정합성 위해 단순 update보다 이력 권장) / **인덱스**(조회 패턴 기반, ESR 순) / **마이그레이션 전략**(무중단이면 expand→백필→contract)
- **(c) API 계약 상세**: req/res 스키마, 에러 코드, HTTP 상태, `Idempotency-Key` 헤더
- **(d) 핵심 메커니즘 — NFR을 *어떻게* 달성**:
  - 트랜잭션 경계 / **동시성**(비관락 vs 낙관락 vs 조건부 atomic) / **멱등성**(키 유니크 제약) / **실패 시 롤백**
- **(e) 대안 비교 + 트레이드오프 + 한계 + 진화 경로** (설계의 필수 — 단일안 결론 금지):

  | 동시성 안 | 장점 | 단점 |
  |---|---|---|
  | 조건부 atomic update (`WHERE balance>=amt`) | 락 없음, 빠름 | 복잡한 로직엔 부족 |
  | 비관락 (SELECT FOR UPDATE) | 직관적, 강한 보장 | 경합 시 처리량↓ |
  | 낙관락 (version) | 읽기 많을 때 유리 | 충돌 시 재시도 로직 필요 |
- **(f) 롤아웃/롤백 계획**: feature flag·마이그레이션 순서·되돌리는 법

> **시니어 포인트**: "비관락 썼습니다"가 아니라 "경합이 드물고 강정합 필요해 비관락, 경합 늘면 조건부 atomic으로 전환" — 택한 이유 + 한계 + 진화 경로까지.

## 3. 분산 트랜잭션 / Saga (적립금·주문이 다른 서비스/DB일 때)

**문제**: 같은 DB면 `BEGIN; 차감; 주문 INSERT; COMMIT;` 으로 ACID가 원자성·자동 롤백을 공짜로 준다. 서비스/DB가 분리되면 두 DB를 감싸는 하나의 트랜잭션이 없어 "돈은 빠졌는데 주문은 없음"이 가능.

### 결정 가이드

```
적립금·주문이 같은 DB?  ── yes ──▶  로컬 ACID 트랜잭션. 끝. (Saga 금지)
        │ no (분리 강제됨)
        ▼
즉시 강한 일관성이 절대 필요?  ── yes(희귀) ──▶  합치거나 2PC
        │ no (최종 일관성 OK)
        ▼
     Saga
   ├ 단계 적고 단순  →  Choreography (이벤트 구독/반응, 중앙 없음)
   └ 단계 多/분기 복잡 →  Orchestration (중앙 조정자 — 추적·디버깅 쉬움)
```

> **right-size 먼저**: 분산 트랜잭션은 서비스 분리의 *대가*. "정말 나눠야 하나?"부터 점검. 같은 DB로 둘 수 있으면 로컬 ACID로 끝낸다.

### 후보 비교

| 안 | 방식 | 일관성 | 트레이드오프 |
|---|---|---|---|
| 2PC (XA) | 코디네이터 prepare→commit | 강한(즉시) | prepare 동안 락 점유, 코디네이터 SPOF, 지연↑, 가용성↓. MSA에서 거의 안 씀 |
| **Saga** | 로컬 트랜잭션 체인 + 실패 시 보상 | 최종(eventual) | 격리성(I) 없음 → 중간 상태 노출. 보상 로직 직접 작성 |
| 분리 안 함 | 한 DB 로컬 트랜잭션 | 강한 | 서비스 결합 |

### Saga 핵심 개념
- **롤백이 아니라 보상(compensation)**: 이미 커밋됐으니 DB 롤백 불가. "차감"을 되돌리는 건 "환불"이라는 새 forward 액션.
- **최종 일관성**: 차감만 되고 주문 없는 상태가 잠깐 실재하지만, 보상이 끝나면 일관된 상태로 수렴.

### 적립금+주문 Orchestration 흐름

```
성공: 1.주문 PENDING (보상:취소) → 2.적립금 차감(보상:환불) → 3.주문 CONFIRMED
실패(2단계 잔액부족): 1✅ 2❌ → 보상 cancelOrder(1) → 주문 CANCELLED, 적립금 그대로
실패(3단계 확정실패): 1✅ 2✅ 3❌ → 역순 보상 refundPoint(2) → cancelOrder(1)
```

보상은 항상 **역순**.

### 진짜 어려운 부분 (신뢰성 3종 + α)
1. **Dual-write 문제 → Transactional Outbox**: "로컬 커밋 + 이벤트 발행"을 원자적으로. 같은 트랜잭션에서 `outbox` 테이블에 INSERT → 별도 릴레이/CDC가 발행. (이벤트 유실 방지)
2. **At-least-once 전달 → 멱등성 필수**: 중복 도착해도 1회만 처리(처리한 `eventId`를 유니크 제약).
3. **격리성(I) 없음 → semantic lock**: 중간 상태 노출. `PENDING` 상태 표시로 "처리 중"임을 알림.
4. **보상 불가 액션 → pivot transaction**: 못 되돌리는 액션(보낸 메일/PG 승인)은 pivot 이후로 배치. pivot 넘으면 보상 대신 재시도로 끝까지 전진.

**한 줄 본질**: 분산 환경에선 원자적 롤백이 불가능하니, 여러 로컬 커밋을 이어 붙이고 + 실패하면 의미적 취소(보상)로 되돌려 + 최종적으로 일관시킨다. 떠받치는 게 **Outbox + 멱등성 + semantic lock**.

## 4. 나머지 단계 상세

설계 이후 단계를 3개 묶음으로 나눠 푼다. 관통 예시는 동일하게 "적립금 구매".

### A. 구현 준비 (작업 분해 → 테스트 리스트업)

#### 작업 분해
**본질**: 설계를 **"단독으로 머지·배포·롤백 가능한 작은 단위"** 로 쪼갬.
- **의존 순서로 배열**: 스키마/마이그레이션 → 레포지토리 → 서비스 로직 → API → 외부 연동/이벤트
- **세로 슬라이스 우선**: "동작하는 얇은 기능 한 줄"씩 쪼개 각 PR이 단독으로 의미를 갖게 (가로 레이어로만 쪼개면 다 합쳐야 동작 → 중간 배포 불가)
- **계약(API contract) 먼저 픽스** → 프론트/타팀 병렬 작업
- **flag 경계 결정**: 어디서 미완성을 가릴지

> **시니어 포인트**: 기준 두 개 — "PR이 리뷰 가능한 크기인가" + "이 단위만 단독 배포돼도 시스템이 안 깨지나". 큰 PR 하나는 리뷰·롤백·충돌을 동시에 악화.
> **적립금 예시 분해**: ① `point_transaction` 스키마+잔액조회 → ② 차감 로직(멱등) → ③ 주문 연계 → ④ 환불/보상 → ⑤ flag on. 각 단계 독립 머지 가능.

#### 테스트 리스트업
**본질**: 코드 짜기 전 **"무엇을 검증할지"** 를 명세에서 뽑음. 원천 = 인수조건 + 기능정의의 예외흐름.
- **케이스 출처**: 해피패스 + 경계값 + 예외/실패 + 동시성/멱등 + 권한
- **피라미드 배치 결정**: 어느 케이스를 단위/통합/e2e로 (비용·속도 trade-off)
- 케이스마다 **"이 테스트가 어떤 회귀를 막는가"** 한 줄

> **시니어 포인트**: 커버리지 숫자가 아니라 "명세의 어느 줄을 지키는 테스트인지". mock 호출 단언이 아니라 **관찰 가능한 최종 상태** 검증.
> **적립금 예시 케이스**: 잔액충분→차감+주문생성 / 잔액부족→거부+미차감 / 동일 멱등키 2회→1회만 / 주문생성 실패→차감 롤백 / 동시 2요청→이중차감 없음 / 만료분 사용 불가.

### B. 구현 & 검증 (구현 → 테스트 → PR·리뷰·CI → e2e)

#### 구현
**본질**: 설계+테스트리스트를 코드로. TDD면 실패테스트 → 최소구현 → 리팩터.
- feature 브랜치, **외과적 수정**(요청 범위만), 미완성은 flag, 기존 스타일 따름
- 추상화는 **2번 이상 반복될 때만**. 한 번 쓸 코드에 추상화 금지

> **시니어 포인트**: 구현 중 설계와 어긋나는 게 보이면 코드 강행이 아니라 **설계로 피드백**(단방향 원칙). 즉흥 결정이 명세-코드 괴리를 만듦.

#### 테스트 진행 (단위·통합)
**본질**: 리스트업한 케이스를 자동화 테스트로 돌려 green 확인.

| 종류 | 검증 | 특징 |
|---|---|---|
| 단위 | 순수 로직·분기 | 빠름, mock 최소 |
| 통합 | DB·외부연동 포함 실제 흐름 | 인메모리 DB 등, 신뢰도↑ |

- **테스트 더블 구분**: stub(반환값만) / mock(상호작용 단언) / fake(경량 구현). 과한 mock은 구현에 결합돼 리팩터 시 깨짐.

> **시니어 포인트**: 관찰 가능한 최종 상태를 검증. 내부 호출 순서·횟수 단언 같은 **깨지기 쉬운 테스트** 지양.

#### PR·코드 리뷰·CI (게이트)
**본질**: 머지 전 최종 관문 = 자동 검증(CI) + 사람 검증(리뷰).
- **CI**: 빌드·타입체크·테스트·린트·정적분석 — green 필수
- **리뷰 받는 쪽**: 맹목 수용도 방어도 아닌 **기술 검증**. 틀린 지적엔 근거로 반박
- PR 설명에 **배포영향 체크리스트**(마이그레이션·env·설정 변경) 명시

> **시니어 포인트**: 리뷰는 스타일 지적이 아니라 **"이 변경이 못 막는 케이스 + blast radius"** 를 본다. 작은 PR일수록 리뷰 품질↑.

#### e2e 테스트 (staging)
**본질**: 배포 전 환경에서 전체 흐름을 실제에 가깝게 검증.
- 느리고 비싸므로 **핵심 시나리오만**(결제 happy path + 주요 실패)
- 단위/통합이 못 잡는 **통합 지점**(외부 PG, 큐, 인증) 검증

> **시니어 포인트**: e2e로 모든 걸 잡으려 하지 않음(피라미드 꼭대기는 얇게). **staging ≈ prod 정합성**이 신뢰의 전제 — 환경이 다르면 통과가 거짓말이 됨.

### C. 배포 & 운영 (배포 → 모니터링 → 롤백 → 회고)

#### 배포
**본질**: 검증된 아티팩트를 환경에 단계적 반영. dev → staging → prod 승격.

무중단 배포 3대 관심사:

| 전략 | 방식 | 특징 |
|---|---|---|
| Rolling | 인스턴스 순차 교체 | 기본, 구·신 잠깐 공존 |
| Blue-Green | 새 버전 통째로 띄우고 전환 | 즉시 롤백, 리소스 2배 |
| Canary | 트래픽 일부→점진 확대 | 위험 최소, 관측 필요 |

+ **마이그레이션 expand-contract** + **flag로 배포≠출시 분리**.

> **시니어 포인트**: "어떻게 되돌리나"가 배포의 *전제*. 스키마와 코드의 **배포 순서**가 무중단의 핵심.

**마이그레이션 expand-contract** — 무중단 배포에선 구·신 코드가 공존하므로 스키마 변경은 단계 분리:
1. **Expand**: 새 컬럼/테이블 추가 (구 코드도 안 깨지게 nullable·기본값)
2. 신 코드 배포 (양쪽 읽기 호환)
3. 데이터 백필
4. **Contract**: 구 컬럼 제거 (모든 인스턴스가 신 코드 된 후)

→ "컬럼 drop + 코드 배포 동시"는 배포 중 구버전 인스턴스가 죽는 사고로 직결.

#### 모니터링
**본질**: 배포 후 시스템 건강 신호를 본다.
- 관측 3종: **메트릭**(에러율·P95 지연·자원) · **로그** · **트레이싱** + 알람
- 배포 직후 대시보드 주시. SLI/SLO 기준

> **시니어 포인트**: 기술 지표(500 에러)뿐 아니라 **도메인 지표**(이중 차감 건수, 잔액 음수 발생, 차감 성공률)를 본다. 알람은 증상이 아니라 **사용자 영향** 기준으로.

#### 롤백 실행
**본질**: 이상 감지 시 빠르게 되돌림.
- **flag off**(가장 빠름·재배포 불필요) → **이전 이미지 재배포** → **blue-green 전환**
- 마이그레이션이 contract(구 컬럼 제거)까지 갔으면 롤백 어려움 → 그래서 expand-contract로 단계 분리

> **시니어 포인트**: **롤백 결정 기준**(어느 지표가 어디를 넘으면 되돌린다)을 배포 *전에* 정해둠. "forward-fix(앞으로 고침) vs rollback" 판단도 미리.

#### 회고 + 정리
**본질**: 사이클을 닫고 부채를 남기지 않음.
- **정리**: 안정화된 flag·임시 코드·deprecated 컬럼(contract) 제거
- **포스트모템**(인시던트 있었으면): 비난 없는 원인분석 + 재발방지 액션
- 문서/CHANGELOG 갱신

> **시니어 포인트**: flag·임시코드를 안 치우면 **다음 사이클의 부채**. "정리"를 막연한 다짐이 아니라 **티켓으로 명시**.

## 5. 관통 원칙 3개

1. **단방향 흐름** — 요구사항 → 기능 → 설계는 위에서 아래로만. 아래에서 빈틈 발견 시 위로 올라가 메우고 내려온다.
2. **right-size** — 작은 변경에 전 단계를 다 밟지 않는다. 변경의 위험도(비가역성·blast radius)에 비례해 design doc / 테스트 / 배포 전략의 깊이를 조절. "이건 design doc 필요 / 이건 PR 설명으로 충분"을 판단하는 게 시니어.
3. **롤백 가능성을 설계에 내장** — 배포·마이그레이션·flag 모두 "잘못됐을 때 빠르게 되돌리는 법"이 기준. 롤백은 사이클 끝에 *생각하는 것*이 아니라 배포의 *전제*.
