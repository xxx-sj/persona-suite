# Auditing Production Risks

## 개요

**구현이 끝난 변경분을 [checklist.md](./checklist.md) 기준으로, 독립 컨텍스트의 서브에이전트가 감사한다.** 체크리스트는 설계 페르소나들이 설계 시점에 잡았어야 할 리스크의 **2차 안전망** — 감사의 목적은 "설계에서 잡았다고 믿는 것"이 코드에 실제로 존재하는지 확인하는 것이다.

체크리스트의 역할은 3층이다: **기준**(8개 카테고리 전부 판정 의무 — 커버리지 하한 보장) + **앵커**(항목은 전수가 아니라 카테고리 본질의 예시 — 구체 질문이 증거 탐색을 유도) + **탈출구**(목록 밖 발견은 `목록외` 라벨로 — 천장 효과 방지).

왜 독립 서브에이전트인가 — 구현을 막 끝낸 메인 컨텍스트는 "이 코드가 왜 맞는지"의 서사를 이미 갖고 있어 자기 감사가 무죄 추정으로 기운다(거짓 통과). 격리가 줄여주는 건 이 **작성자 편향**과 긴 컨텍스트의 혼동이고, **무근거 위반 주장(false positive)은 격리로 줄지 않는다** — 그건 다음 구조로 막는다:

1. **작성자 서사 미전달** — 감사자는 diff 와 코드만 받는다. 구현 의도·정당화·설계 문서의 "한계" 섹션은 주지 않는다.
2. **근거 인용 의무** — 위반/통과 판정엔 `파일:라인` + 코드 인용 필수. 없으면 확인불가로 강등.
3. **항목별 사용자 게이트** — 위반의 수용/기각은 사용자가 항목별로 (배치 수용 금지. [PRINCIPLES §1](../../PRINCIPLES.md)과 동일 — 결정은 사용자).
4. **bounded 재감사** — 수정 후 재감사는 1회 상한. 무한 순환은 종료 조건이 판정자 판단에 떠서 false positive 에 끌려다닌다.

## 사용 시점

- 피처 구현+테스트(7~8단계)가 끝나고 PR 올리기 **전** (라이프사이클 8.5)
- 재화·외부 연동·동시성·큐·마이그레이션을 건드린 변경이면 **skip 금지**
- 가벼운 tier(문구·단순 조회 등)고 위 영역 비접촉이면 skip 가능 (right-size)

**쓰지 않을 때:**
- 코드 품질·스타일·구조 → 코드 리뷰 (9단계, 예: code-review 류)
- 테스트 케이스 충분성 → `validating-test-lists` (6b, 작성 전) / 테스트 감사 도구 (작성 후)
- 스펙 적합성(AC 충족) — 요구사항 문서 AC 대조는 구현 단계의 몫

## 철칙

**근거 없이는 판정 없음. 서사 없이 코드만. 결정은 사용자.** 모든 위반/통과 판정에는 `파일:라인` 근거가 따라붙고, 감사자는 작성자의 설명이 아니라 코드 자체를 읽으며, 위반의 수용/기각은 사용자가 항목별로 정한다.

## 절차

### 1. 감사 범위 산정
- 변경분 확정: base 브랜치 대비 `git diff` → 변경 파일 목록.
- 피처 한 줄 요약 — "무엇을 추가/변경했나"만 (의도 설명·정당화 아님).
- right-size 판단: 위 "사용 시점" 기준으로 skip 여부. skip 하면 그 사실과 이유를 기록하고 끝.

### 2. 프로젝트 특정 항목 주입
- 프로필(`.claude/persona-profile.md`)의 `crosscutting`/`capacity` 가 있으면 체크리스트에 프로젝트 행으로 추가 (예: 재화 정책, 외부 연동 목록). 없으면 제네릭 체크리스트만으로 진행 (가정·하드코딩 금지).

### 3. 독립 감사 dispatch (단발)
읽기 전용 서브에이전트(Explore/general-purpose 류)에 전달:
- 피처 한 줄 + 변경 파일 목록(diff)
- [checklist.md](./checklist.md) 전문 + (있으면) 프로필 주입 항목
- 아래 판정 양식 + 근거 인용 의무 + "항목은 앵커 — 카테고리 본질로 판정, 목록 밖 리스크는 `목록외`로 보고"

**전달 금지**: 구현 의도/정당화, 설계 문서의 "한계/트레이드오프" 섹션, "이건 이래서 안전함" 류 사전 변호. 감사자는 변경 파일과 그 주변 코드(스키마·인덱스·호출부)를 직접 읽고 판정한다.

| 판정 | 의미 |
|---|---|
| **위반** (`CRITICAL`/`WARN`) | 체크 항목이 깨졌다 — `파일:라인` + 코드 인용 + 깨지는 시나리오 한 줄 |
| **통과** | 해당 항목을 코드가 막고 있다 — 막는 코드 `파일:라인` 인용 |
| **해당없음** | 이 변경엔 그 카테고리가 없다 — 한 줄 이유 |
| **확인불가** | 판정에 필요한 파일/정보를 못 읽었다 — 뭘 못 봤는지 명시 |
| **목록외** (`CRITICAL`/`WARN`) | 체크리스트 밖 리스크 발견 — 같은 근거 의무. 자유 발상이라 false positive 확률이 높음을 사용자가 알도록 라벨 명시 |

### 4. 리포트 검수 (기계적)
리포트를 받으면 **근거 형식만** 검사한다: 위반/통과/목록외인데 `파일:라인` 인용이 없으면 확인불가로 강등. 내용 재판정은 하지 않는다 — 메인 컨텍스트의 재판정은 작성자 편향을 다시 들이는 것.

### 5. 항목별 사용자 게이트
위반·목록외 항목을 심각도순으로 하나씩 제시 → 사용자가 항목별 수용/기각. "다 고쳐" 식 배치 요청이 와도 `CRITICAL` 은 항목별 확인을 받는다. 확인불가 항목은 감사자가 못 읽은 부분을 메인이 보충 확인해 해소한다. **수용된 목록외 항목은 checklist.md 편입 후보로 제안** (역주입 — 단 편입 여부도 사용자 결정).

### 6. 수정 → 재감사 (1회 상한)
- 수용 항목 수정 후, **새** 서브에이전트에 수정 diff + 원 위반 목록만 전달: "각 위반이 해소됐나 + 수정이 새 위반을 만들었나".
- 재감사는 1회. 이후에도 남는 항목은 사용자 판단으로 종료하고 9단계(PR·코드리뷰)에 메모로 인계한다.

## Quick Reference

```
변경 diff + 피처 한 줄    →  읽기 전용 서브에이전트 (작성자 서사 미전달)
checklist + 프로필 주입   →  위반/통과/해당없음/확인불가/목록외 (+ 파일:라인 근거)
근거 없는 판정           →  확인불가로 강등 (기계적 검수)
위반·목록외              →  사용자 항목별 수용/기각 → 수정 (목록외 수용분은 체크리스트 편입 후보)
수정분                  →  재감사 1회 (해소 + 신규 위반만) → 종료, 9단계로 인계
```

## 흔한 실수

- **메인 컨텍스트에서 자기 감사한다.** 방금 구현한 컨텍스트는 자기 코드 무죄 추정 — 반드시 독립 서브에이전트로.
- **구현 의도/정당화를 같이 넘긴다.** "조건부 업데이트라 안전함" 같은 사전 변호가 들어가면 감사가 아니라 확인 도장이 된다.
- **근거 없는 판정을 받아준다.** 안 읽은 파일에 대한 "인덱스 없음" 단정 — `파일:라인` 인용 없으면 확인불가.
- **체크리스트를 천장으로 쓴다.** 항목은 앵커지 전수가 아님 — 카테고리 본질로 판정하고 목록 밖은 `목록외`로 보고.
- **리포트를 배치 수용한다.** 위반 전부를 한 번에 반영하면 false positive 가 코드를 망친다 — 항목별 독립 검증.
- **재감사를 반복한다.** 라운드마다 새 지적은 LLM 리뷰어의 기본 성질 — 1회 상한 후 사람 판단으로 종료.
- **가벼운 변경에 풀 감사를 돌린다.** 문구 수정에 8 카테고리는 세리머니 — right-size skip.
