# 프로덕션 리스크 체크리스트 (감사 입력)

> `auditing-production-risks` 감사 서브에이전트의 판정 기준. **설계 페르소나([TECH][SEC][DATA][SRE][OPS])가 설계 시점에 잡았어야 할 것들의 2차 안전망**이다 — 특정 카테고리에서 위반이 반복 검출되면 설계 단계 질문뱅크를 보강하라는 신호.
>
> **항목은 전수 목록이 아니라 카테고리 본질의 앵커다.** 감사자는 카테고리(예: 동시성)의 본질로 판정하고, 항목은 예시/출발점. 카테고리 안이지만 항목에 없는 발견은 그 카테고리로, 카테고리 밖 발견은 `목록외` 판정으로 보고한다 (SKILL.md 판정표 — 같은 근거 의무).
>
> 이 목록은 완전하지 않은 게 전제다 — 운영(라이프사이클 12~13)에서 목록 밖 사고를 만나면 **이 파일에 역주입**한다 (회고 루프). 목표는 태어날 때 완전이 아니라 이 프로젝트의 실제 사고 분포로 수렴.
>
> 프로젝트 특정 항목(재화 정책·외부 연동 목록·큐 이름)은 여기 추가하지 않는다 — 프로필 `crosscutting`/`capacity` 에서 감사 시점에 주입 (이식성 원칙).

**심각도**: `CRITICAL` = 돈·정합성·보안·데이터 손실 / `WARN` = 성능·운영·관측.

## 1. 돈/재화 정합성 (CRITICAL)
- 이중 과금·이중 지급 — 더블탭/클라이언트 재시도/웹훅 중복 수신이 같은 효과를 2번 일으키나? (멱등키·unique 제약·조건부 업데이트 증거)
- 외부 결제 API 멱등성 — 타임아웃 후 재시도에 멱등키를 전달하나?
- 부분 실패 — 외부 성공 + 내부 실패 시 보상 경로(보상 트랜잭션/outbox/수동 개입 기록)가 있나?
- 장부 대사 — 우리 기록과 외부(PG 등) 기록을 맞춰볼 수단이 있나?
- 음수 잔액 — 잔액 검증이 read-then-write 인가, 조건부 원자 차감인가?

## 2. 동시성 (CRITICAL)
- read-modify-write 경합 — 조회와 수정 사이에 끼어들면 깨지는 invariant 가 있나? (조건부 atomic / 낙관락 증거)
- 중복 생성 — 애플리케이션 검사만인가, unique 인덱스가 받치나?
- 다중 인스턴스 — in-memory 상태/락에 의존하는 코드가 생겼나?

## 3. 멱등성/재시도 — 비동기 (CRITICAL)
- 큐 작업 at-least-once 재처리 — 같은 작업이 2번 와도 부수효과(지급·발송)가 1번인가?
- 웹훅 중복·순서 역전 — 중복 수신/늦은 도착을 처리하는 경로가 있나?
- 실패한 작업의 행방 — 조용히 사라지나(catch 후 무시), DLQ/재시도/알람으로 이어지나?

## 4. 외부 의존 (WARN — 단, 지연 전파가 재화 경로면 CRITICAL)
- 타임아웃 — 신규 외부 호출에 타임아웃이 설정돼 있나?
- 장애 격리 — 외부 다운 시 해당 기능만 죽나, 전체로 전파되나? (서킷브레이커/격리 증거)
- 응답 방어 — 외부 응답 스키마를 무방비로 파싱하나?

## 5. 데이터 (백필 누락·drift = CRITICAL / 성장 = WARN)
- 기존 document — 새 필드가 기존 데이터에 없을 때(백필 전/누락) 코드가 안전한가? (`?? 기본값` / 마이그레이션 증거)
- denormalized drift — 복제 필드의 source of truth 와 갱신 경로가 한 곳인가?
- unbounded 성장 — 무한히 자라는 배열/컬렉션이 생겼나? 조회가 그걸 통째로 읽나?
- 인덱스 — 신규 쿼리를 받는 인덱스가 있나? (데이터 증가로 점진 악화 — 배포 직후엔 멀쩡)

## 6. 성능/자원 (WARN)
- N+1 — 루프 안 쿼리/외부 호출?
- 캐시 — 무효화 누락(stale)? 개인화 값이 공유 캐시에 들어갔나? 스탬피드?
- 메모리 — 대형 payload/전건 로드가 인스턴스 메모리 한계와 충돌하나? (프로필 `capacity` 대조)
- 부하테스트 게이트 재판정 — 요구사항 문서 §5 의 부하테스트 판정(신호: 새 hot path·fan-out·직렬화 지점 추가·동기 외부의존·unbounded 쿼리·capacity 근접·스파이크 예정)이 '필요'인데 수행 계획이 없나? (1차 게이트의 2차 안전망)

## 7. 권한/보안 (CRITICAL)
- 소유자 검증 — 리소스 id 만 알면 남의 것을 조작할 수 있는 경로(IDOR)가 있나?
- 민감정보 — 에러 응답/로그에 토큰·개인정보가 새나?

## 8. 운영/배포 (WARN — 단, 롤백 불가는 CRITICAL)
- 알람 — 이 피처가 깨지면 유저 제보보다 먼저 알 수 있나? (메트릭/알람/로그 증거)
- 롤백 — 이 배포를 되돌릴 수 있나? (expand→contract 순서, 파괴적 마이그레이션 여부)
- 점진 배포 — blast radius 큰 변경이 플래그/단계 없이 빅뱅인가?
