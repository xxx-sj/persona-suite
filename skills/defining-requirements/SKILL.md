# defining-requirements

대략 정해진 피처를 6개 시니어 페르소나의 압박 질문(티키타카)으로 **①요구사항 → ②기능정의 → ③기술설계** 단방향으로 깎아 요구사항 문서로 확정한다. 사용자가 직접 답하며 정의하므로 결과물(문서)과 학습(프로세스 체화)을 동시에 얻는다.

## 핵심 원칙

1. **실제 피처 앵커 강제** — 가상 피처 금지. 사용자가 실제로 곧 만들/고칠 피처여야 한다.
2. **능동적 리뷰어 — 결정만은 사용자 (피드백 모드)** — 이건 개발 단계 도구라 *피드백 모드*다: 사용자가 제안 → 페르소나가 빠진 곳·약한 곳 지적 + 검증 + (도출) 초안 제안. **결정(정책·우선순위)만은 대신 내리지 않는다** (정책엔 코드에 박힌 정답이 없음). "답을 통째로 주지 않는다"는 *결정을 대신 안 한다*는 뜻이지 피드백·검증·제안을 안 한다는 게 아니다. 순수 Socratic(백지 도출)은 학습용(`architecture-interview`)이며 이 스킬이 아니다.
3. **단방향 + 역류** — 흐름은 ①→②→③ 위에서 아래로만. 아래 단계에서 "이 경우 어떻게?"가 나오면 즉흥 결정하지 말고 한 칸 위로 올라가 메우고 내려온다.
4. **개념은 본질에서 도출 (항상)** — 도메인 개념을 만나면 "코드에 있으니 규칙"이라 단정하지 않는다. *그게 뭔지(카테고리)* 를 파악하고(모르면 "이게 뭐야?"부터 질문), 그 카테고리의 본질에서 관심사를 도출한다. 예: 루비=재화→곧 돈→원자성·이력·음수방지·멱등. 출처는 `[선언]`(문서가 '반드시' 명시) / `[도출]`(본질에서) / `[관찰]`(코드서 그러더라 — 강제·일관성 미확인) / `[추정]` 로 구분한다. **프로필 작성 때뿐 아니라 핑퐁 중 새 개념을 만날 때도 매번** 이렇게 한다.
5. **관심사는 적되 솔루션은 처방하지 않는다 (right-size)** — crosscutting·NFR은 *질문/관심사 + 스케일 축* 으로 적고 특정 솔루션을 기본값으로 깔지 않는다. ❌ "이벤트 발행=outbox 써라" → ⭕ "이 이벤트 유실되면 뭐가 깨져? 허용이면 commit 후 단순 발행(T0), 불가(재화·정산)면 outbox(있으면 재사용)". 해법은 항상 프로필 `capacity`(현재 규모)에서 **가장 단순한 것부터**; 무거운 패턴(outbox·분산락·saga)은 반사적으로 깔지 않고 규모가 요구할 때 escalate. 즉 crosscutting은 "써라"가 아니라 "이건 챙겼어? + 지금 규모엔 이게 최단순".
6. **챕터 단위 자기 점검 루프** — 각 페르소나/단계를 확정하기 *직전*, 사용자 답을 받았어도 한 번 더 **독립적으로** "이 챕터에 더 필요한 게 없나?"를 스스로 audit한다. 빠진 게 있으면 피드백(추가 압박), 없으면 확정. 종료 완성도 패스(G3)의 *챕터 단위* 버전 — 끝에 몰지 말고 매 챕터마다.

## 시작 전: 프로젝트 프로필 로드

페르소나가 "이 프로젝트의 용량 제약/관례"로 압박하려면 프로젝트 정보가 필요하다.

1. `<repo>/.claude/persona-profile.md` 를 찾는다.
2. **있으면** 로드해 페르소나 압박에 주입한다 (재질문 없음).
3. **없으면 (새 프로젝트)** 만들고 시작한다. 프로필은 두 부분이고 *채우는 방식이 다르다*:
   - **(a) 객관 사실 — upfront, auto 추론 + 빠른 확인**: stack(package.json) / output_path·doc_dir(CLAUDE.md) / handoffs(존재 확인) / capacity(메모리·CLAUDE.md, **모르면 반드시 질문** — right-size 기준). 검증 가능한 사실이라 추론 후 확인으로 충분.
   - **(b) crosscutting — upfront 채우지 않는다 (lazy + 피드백 co-define)**: 판단·도출이라 auto-fill해 던지면 사용자가 *감사만* 하게 됨(스킬 철학 위반). 대신 **핑퐁 중 피처가 개념을 건드릴 때 그 자리서** [SEC]/[TECH]가 원칙4·5로 *도출 초안*을 제안 → 사용자 피드백 → 결정 → crosscutting에 `[선언]/[도출]/[관찰]/[추정]` 라벨로 누적. 프로필은 객관 사실로 시작해 피처를 돌수록 자란다.
   - **객관 사실만** `<repo>/.claude/persona-profile.md` 로 저장(프로젝트 지식 → git tracked) 후 진행. crosscutting은 빈 채로 시작해 누적.

프로필 스키마:
```yaml
project:      { name, stack }
capacity:     # CPU/메모리/단일성/데이터량 — [TECH][SEC] 압박에 사용
conventions:  { output_path, doc_dir }   # 출력 위치
handoffs:     { db_design, test_list, plan }   # 존재하는 것만 (제안 전 존재 확인)
crosscutting: []  # upfront 비움. 피처가 개념 건드릴 때 [SEC]/[TECH]가 co-define 누적
```

프로필이 없어도 페르소나는 "이 프로젝트 용량 제약이 뭐야?"처럼 **물으면서** 동작한다 (가정 금지).

## 페르소나 로드

이 스킬 디렉토리의 `personas.md` 를 읽어 6개 페르소나(`[PM] [REV] [TECH] [SEC] [OPS] [QA]`)의 역할·질문뱅크·트리거를 로드한다. `[REV]` 가 메인 진행자로 흐름을 끌고, 트리거에서 다른 페르소나를 호출한다. 사용자가 "지금 [PM]만"처럼 특정 페르소나를 개별 호명하면 그 페르소나만 돌린다.

## 진행 절차

```
0. 앵커 + right-size 게이트
     - "어떤 실제 피처? (가상 금지)" + 신규/수정 판별
     - 변경 무게(비가역성·blast radius) → tier 결정 (아래)
     - 수정이면: 현행 코드 조사 → AS-IS 맵 산출 (질문 한 줄이 아닌 일급 step)
1. [PM]  ① 요구사항 (비즈니스: 왜·누구·성공지표·범위/비범위·제약·NFR)
2. [REV] ② 기능정의 (블랙박스 동작·예외/대안흐름을 관찰가능한 결과로·상태전이·입출력)
3. [TECH] ③ 기술설계(고수준: 영향범위·컴포넌트·데이터모델·API계약·NFR달성·관측성·대안비교·롤아웃/롤백)
     - 스키마 상세 설계로 깊어지면 → 핸드오프 (db_design 프로필 타깃)
     - 빈틈 발견 → ②/① 역류·복귀
4. [SEC] 횡단 (재화·결제·권한·PII 트리거 시)
5. [OPS] 운영 흐름 (등록/수정/취소/복구·admin UI·장애 수동개입)
6. [QA]  AC (②의 관찰가능결과 → 검증가능 AC·엣지). 테스트 리스트 검증은 핸드오프(test_list 타깃)
7. [REV] 종료 완성도 패스 — ①②③ 가로질러 "아직 안 정해진 게 뭐냐" 훑기
8. 산출 → 프로필 output_path 의 요구사항 문서 (output-template.md 양식) → writing-plans 핸드오프
```

### right-size tier (게이트 0에서 결정)

- **사소** (변수·문구·자명) → 스킬 건너뜀 또는 `[REV]` 빠른 패스 1회.
- **중간** (단일 모듈·가역) → `[PM] [REV] [QA]`.
- **무거움** (비가역·재화·마이그레이션·다서비스) → 풀 6종 + 분산 트랜잭션(Saga/Outbox/보상) 깊이.

작은 변경에 전 단계를 다 밟지 않는다. 무게에 비례해 깊이를 조절한다.

## 피드백 모드 (개발 단계 = 능동적 리뷰어)

이 스킬은 학습용 Socratic(백지에서 도출)이 아니라 **개발 단계의 피드백 모드**다. 순수 Socratic 학습은 `architecture-interview`가 따로 있다.

- **사용자가 제안 → 페르소나가 리뷰**: 빠진 곳·약한 곳 지적 + 검증(코드/현황 대조) + 도출 초안 제안.
- 초안은 **"확정 사실"이 아니라 "초안 — 피드백 줘"** 로 던진다 (과장 라벨 금지).
- 사용자가 **막혔을 때만** 2~3개 각도를 더 제시 (떠먹이기 방지).
- **결정(정책·우선순위)만은 항상 사용자가 커밋.** "답을 통째로 안 준다"=결정 대신 안 함이지 피드백·검증 안 함이 아니다.
- **용어**: 확정된 결정은 "**확정**"으로 표기한다. "락(lock)"은 쓰지 않는다 — 기술적 락(분산락·DB락·atomic lock)과 혼동된다.

## 출력

이 스킬 디렉토리의 `output-template.md` 양식으로 요구사항 문서를 작성한다. 위치는 프로필 `conventions.output_path`. 테스트 리스트는 기본 AC까지만 내고 "→ 테스트 리스트는 test_list 핸드오프" 포인터를 둔다. 무거운 tier면 테스트 리스트 초안까지 끌고 가 검증으로 넘긴다.

## 핸드오프 (재구현 금지)

| 닿는 지점 | 넘기는 곳 |
|---|---|
| 상세 스키마/인덱스 설계 | 프로필 `handoffs.db_design` |
| 테스트 케이스 리스트 검증 | 프로필 `handoffs.test_list` |
| 구현 계획 | 프로필 `handoffs.plan` (없으면 writing-plans 류 계획 스킬) |

핸드오프 타깃의 **실제 경로/이름은 프로젝트 프로필에서만** 가져온다 (플러그인에 특정 프로젝트 경로를 박지 않는다). 제안 전에 존재 여부를 확인한다 (타 프로젝트엔 없을 수 있음).

## 금지

- 가상 피처로 진행하기.
- 정책·우선순위를 페르소나가 대신 결정하기.
- 프로젝트 specifics(용량 숫자·경로)를 가정으로 하드코딩하기 — 프로필/질문으로 얻는다.
- 이미 도구가 있는 일(코드 리뷰·테스트 리스트 감사·상세 DB 설계)을 페르소나로 재구현하기.
