# 요구사항 문서 출력 양식

`defining-requirements` 산출물. 프로필 `conventions.output_path` 에 `<도메인>/<피처>.md` 로 저장. 해당 섹션이 비면 채워졌는지 종료 완성도 패스에서 확인한다. 피처에 해당 없으면 "해당 없음"으로 명시(빈칸 방치 금지). 완성 깊이의 기준은 [`examples/defining-requirements-example.md`](../../examples/defining-requirements-example.md) 참고.

# <피처 이름>

## 1. 개요 (Context)
- 문제(Why): <지금 뭐가 안 되는가>
- 해결책 요지(What): <무엇을 한다>
- 배경: <발생 맥락>
- 비범위(Non-scope): <이번에 안 하는 것 — 명시해야 구현 세션이 추측으로 채우지 않는다. 없으면 "없음" 명시>
- 성공지표: <지표 + 현재값 → 목표값. 계측 수단 명시 — 계측이 없으면 계측 추가가 범위(§4)에 들어간다>
- **사후 판정 (게이트 — 생략 금지)**: 판정 기준 <지표가 Y면 성공/미달> · 판정 예정일 <배포 +N일 — 배포 시 절대 날짜(YYYY-MM-DD)로 치환해 스캔 가능하게> · 판정 방법 <어느 메트릭/대시보드로 조회>.
  - 예정일 도래 시 실측과 대조해 **유지/개선/제거**를 결정한다 (라이프사이클 12 — 수행은 프로젝트 몫, 예: 운영 리포트 루틴에 얹기). 성공지표를 정의만 하고 소비하지 않는 것을 막는 장치.

## 5. 비기능 요구 (NFR)
- 정합성 / 동시성 / 멱등성 / 성능(P95·부하) / 가용성 / 관측성 / 감사 — 해당 항목만, 각 한 줄.
- **부하테스트 판정 (게이트 — 생략 금지)**: 필요/불필요 + 걸린 신호 + (필요 시) 측정 수치(P95·RPS 목표).
  - 신호: 새 hot path · 요청 증폭(fan-out) · 직렬화 지점 추가(락/트랜잭션) · 동기 외부 의존 추가 · unbounded 데이터 쿼리 · capacity 임계 근접 · 스파이크 예정.
  - 판정 입력은 추측 트래픽이 아니라 **실측**(현재 메트릭)과 프로필 `capacity` 대조. '필요'면 수행은 라이프사이클 10.5(프로젝트 몫).
