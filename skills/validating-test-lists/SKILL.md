# Validating Test Lists

## 개요

**테스트 목록은 피처 정의를 기준으로 검증한다.** 피처가 요구하는 동작·business rule·state transition·경계·invariant 에서 "기대 동작 목록(expected-behavior inventory)"을 도출하고, 사용자가 제안한 각 테스트를 거기에 비춰 판정한다. 모든 판정에는 **근거**가 붙는다 — 어떤 피처 규칙에서 나왔는지, 또는 어떤 기존 테스트와 겹치는지.

기준은 피처지만, **연관된 기존 코드·테스트를 보조로 확인**한다: (a) 이미 커버된 중복을 걸러내고, (b) 피처가 건드리는 기존 제약/invariant 에서 놓친 경계를 찾고, (c) 실현 가능성을 점검한다. 주(主)는 피처 정의, 코드는 보조.

이건 *이미 작성된 테스트 코드 감사*가 아니다 (그건 `test-reviewer`). 구현이 아직 없을 수도 있다 (새 피처 / TDD) — 그래서 기준은 "구현 코드"가 아니라 "피처 정의"다.

## 사용 시점

- 새 피처를 정의하고 테스트 코드를 짜기 **직전**, 케이스 목록이 충분한지 검증
- 사용자가 테스트 목록을 주며 "충분한가? / 빠진 거 없나? / 이 테스트 필요한가?" 물을 때
- TDD fail-first 전, spec 작성 전 케이스 목록 검증

**쓰지 않을 때:**
- 이미 작성된 테스트 감사 → `test-reviewer`
- 테스트 코드 작성 자체 → `test-writer`
- 피처 정의가 없음 → 검증할 기준이 없으니 정의부터 확정한다

## 철칙

**피처 정의 없이는 판정 없음. 근거 없이는 판정 없음.** 피처가 모호하면 그것부터 드러낸다 — 정의되지 않은 피처에는 테스트를 검증할 수 없다. 모든 KEEP/MODIFY/DELETE/ADD 에는 피처 규칙 또는 기존 spec 근거가 따라붙는다.

## 절차

### 1. 피처 정의를 확정하고 기대 동작을 도출
피처 정의에서 다음을 뽑아 **expected-behavior inventory** 를 만든다:

- 핵심 동작과 **business rule**
- **state transition** (예: `review → approved`) 과 전이 조건
- **경계값** (개수 임계, 금액, 시간, 비율)
- **에러 조건 / 거부 경로** (권한, 중복, 부족, 충돌)
- **invariant** (항상 참이어야 하는 것)
- **재화 / 원자성 / 멱등성** 요구 (차감-실패 시 rollback 등)

정의가 모호하면 임의로 채우지 말고 질문한다 (예: "비용이 2~3 루비 — 정확히 몇? 범위로는 단언을 못 박는다").

### 2. 연관 기존 코드·테스트를 보조로 확인
Glob/Grep 으로 (범위 넓으면 Explore 에이전트 dispatch):

- 연관 `*.spec.ts` — **이미 커버된 것**은 다시 제안하지 않는다 (중복 근거로 쓴다)
- 연관 모델/서비스 — 이 피처가 **건드리는 기존 제약/invariant** → 놓친 케이스 또는 도달 불가능 판단
- 기존 패턴 — 같은 종류의 동작이 이미 테스트되는 방식

### 3. 각 제안 테스트를 분류 (근거 필수)
항목마다 판정 하나 + 근거(피처 규칙 또는 기존 spec):

| 판정 | 의미 |
|---|---|
| **KEEP** | 피처의 실제 규칙/동작에 매핑됨; 단언 대상이 옳음 |
| **MODIFY** | 규칙은 맞으나 단언이 약함/틀림 — 경계가 너무 좁거나, 관찰 불가능한 것을 단언 (5번 참조) |
| **DELETE** | 불필요 — 기존 테스트와 중복, 또는 피처 범위 밖, 또는 framework/library 보장을 테스트, 또는 어떤 피처 규칙과도 무관 |
| **CLARIFY** | 피처 정의가 모호해 이 테스트의 옳고 그름을 아직 판정 불가 → 결정해야 할 질문으로 올린다 |

### 4. 빠진 항목 도출 (ADD)
expected-behavior inventory 를 훑어 **제안 테스트가 없는** 동작·경계·에러 경로·invariant 를 ADD 후보로 올린다. 각각 어떤 피처 규칙에서 나왔는지 근거를 단다. **재화/원자성·state-count 경계·중복/권한 가드**를 우선한다 — 사용자가 스펙만 보고 가장 자주 놓치는 곳이다.

### 5. behavior-contract 패스
단언 대상이 관찰 가능한 최종 state 가 아닌 테스트는 거부하고 재작성한다:
- "X 함수가 호출됐다 / mock called" → 결과 **state** 를 단언 (잔액, status, 큐 내용)
- "예외가 전파된다 / error propagates" → 관찰 가능한 결과를 단언 (요청 거부됨, state 불변)
- "호출 순서가 맞다 / call order" → 순서가 아니라 최종 state 를 단언

### 6. 리포트
- **판정 표**: 항목 → 판정 → 근거(피처 규칙 / 기존 spec)
- **ADD 목록**: 빠진 케이스, 피처 근거와 함께, 우선순위순
- **명세 모호점**: 테스트 설계를 막는 결정 사항 — 임의로 채우지 말고 질문

## Quick Reference

```
피처 정의            →  expected-behavior inventory (동작·전이·경계·에러·invariant·재화)
연관 코드/테스트(보조) →  이미 커버된 것(중복) + 건드리는 기존 제약
각 제안 테스트        →  KEEP / MODIFY / DELETE / CLARIFY  (+ 피처 규칙·spec 근거)
빠진 동작·경계        →  ADD (재화/원자성·count 경계·중복/권한 우선)
관찰 불가능한 단언     →  최종 state 로 재작성
모호한 명세          →  플래그 & 질문, 임의로 채우지 않음
```

## 흔한 실수

- **피처 정의 없이 목록부터 판정한다.** 검증할 기준이 없으니 일반론만 나온다. 정의(또는 모호점)부터 확정한다.
- **근거 없이 판정한다.** "이것도 추가하면 좋음" 식 제안 금지 — 어떤 피처 규칙/기존 spec 때문인지 반드시 댄다.
- **연관 기존 테스트를 안 보고 중복을 제안한다.** 기준은 피처지만, 이미 있는 테스트는 코드로 확인해야 걸러진다.
- **피처 범위 밖 테스트를 받아준다.** 피처가 정의하지 않은 동작은 KEEP 이 아니라 DELETE 또는 CLARIFY.
- **"X 호출됨" 식 테스트를 받아준다.** mock-call / exception-propagation / call-order 단언은 behavior contract 위반 — 관찰 가능한 state 로 재작성.
- **모호한 명세를 직접 채운다.** "2~3 루비"나 "approved (rejected 는?)" 는 사용자가 내릴 결정 — 질문으로 드러낸다.
